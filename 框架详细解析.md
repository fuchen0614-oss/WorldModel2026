## 二、当前处于什么模式（重要）

### 2.1 数据模态：当前是 S2L2A

代码现在默认读取 **S2L2A** 模态：Sentinel-2 L2A 级产品，每个样本是 `[4个季节, 12个波段, 264, 264]` 的 int16 数组。训练时：

- 随机抽取 1 个季节（`random_season: true`），得到 `[12, 264, 264]`
- 中心裁剪到 `[12, 256, 256]`（256 能被 patch_size=16 整除）
- 归一化到 `[0, 1]`（除以 10000）

为什么选 S2L2A 而不是更简单的 S2RGB：因为完整 ObsWorld 后续要做多光谱/多模态，编码器从一开始就吃 12 通道，未来不必改输入层。

### 2.2 如何切换到 S2RGB 模式

只需要改**两处配置**（不需要动任何代码）。以单卡配置 `configs/train/stage1_single.yaml` 为例：

```yaml
# 第一处：data.modality
data:
  modality: S2RGB          # 从 S2L2A 改成 S2RGB

# 第二处：模型输入/输出通道（12 → 3）
model:
  encoder:
    in_channels: 3         # 从 12 改成 3
  decoder:
    out_channels: 3        # 从 12 改成 3（必须与 encoder.in_channels 一致）
```

切换后数据形状变化：

| | S2L2A（当前） | S2RGB（切换后） |
|---|---|---|
| 原始 bands | `[4, 12, 264, 264]` int16 | `[4, 3, 264, 264]` uint8 |
| 归一化 | 除以 10000 | 除以 255 |
| 编码器输入 | `[B, 12, 256, 256]` | `[B, 3, 256, 256]` |

归一化逻辑会根据 `modality` 自动选择正确的除数，定义在
[data/datasets/ssl4eo.py](data/datasets/ssl4eo.py) 的 `normalize_image()` 函数里，无需手动改。

> 提示：FSDP 配置 `configs/train/stage1_fsdp.yaml` 同样的两处也要一起改。

```
   │  逐个 tar 流式吐出样本（不需把 2.3TB 读进内存）
   │  多卡时 split_by_node 自动按 rank 分片 → 每卡读不同 tar
   ▼
[解析 zarr.zip]  parse_zarr_sample()
   │  zipfile 打开 → zarr 读 bands/cloud_mask/time/经纬度
   │  random_season 抽 1 季 → 中心裁剪 264→256 → 归一化
   ▼
[组装 batch]  collate_fn()  → DataModule
   │  返回 dict: image[B,C,H,W] + phi(成像条件) + field_mask
   ▼
[编码器]  models/encoders/tiny_vit_encoder.py
   │  PatchEmbed 切 patch → 加位置编码 → 随机遮掉 75% patch
   │  只把可见的 25% patch 过 Transformer → 输出 latent
   ▼
[解码器]  models/decoders/light_decoder.py
   │  把可见 token + 占位 mask token 拼回完整序列
   │  Transformer 解码 → unpatchify 还原成图像 [B,C,H,W]
   ▼
[损失]  models/losses/reconstruction.py
   │  只在被遮住的 patch 上算 MSE（masked_mse）
   ▼
[反向传播 + AdamW 更新]  train/train_stage1_ssl4eo.py
   │  多卡时 FSDP 自动 all-gather 参数 / reduce-scatter 梯度
   ▼
[保存 checkpoint]  汇聚完整权重 → 仅 rank 0 写盘
```

## 四、FSDP 架构详解（核心）

### 4.1 一句话原理

FSDP（Fully Sharded Data Parallel，完全分片数据并行）把模型的**参数、梯度、优化器状态**沿设备维度切成 N 份，N 张卡每张只存 1/N。需要某层做计算时，临时把那一层的完整参数从所有卡 all-gather 凑齐，算完立刻丢弃非本卡的碎片。这是**用通信换显存**——让单卡放不下的大模型能训起来。

> 当前的 4.87M tiny 模型单卡绰绰有余，**现阶段用 FSDP 纯粹是为了把多卡分布式流程跑通、验证正确**，为后续 Stage 2/3 上大模型铺路。这一点在代码注释和本文档里都如实说明。

### 4.2 本项目用的是 FSDP2（fully_shard）

torch 2.12 同时提供两代 API，本项目用更新的 **FSDP2**：

| | FSDP1（旧） | FSDP2 / fully_shard（本项目采用） |
|---|---|---|
| API | `FullyShardedDataParallel(model, ...)` | `fully_shard(model, mp_policy=...)` |
| 分片单位 | flat-parameter（打平成一维） | **per-parameter（DTensor，按参数分片）** |
| 混合精度参数 | `mixed_precision=MixedPrecision(...)` | `mp_policy=MixedPrecisionPolicy(...)` |
| 状态 | 旧，逐步废弃 | torch 2.x 官方未来方向 |

> ⚠️ 这两套 API 不能混用。本项目早期版本曾误用 `fully_shard(model, mixed_precision=MixedPrecision(...))`——这是把 FSDP2 的函数和 FSDP1 的参数名/类型混在一起，会直接报错。当前代码已修正为 FSDP2 正确写法，并在本机 torch 2.12 实测通过。

### 4.3 多卡训练的完整执行链

以 `torchrun --nproc_per_node=4` 为例，启动后每张卡是一个独立进程，各自执行同一份 `train_stage1_ssl4eo.py`：

```
torchrun 注入环境变量 RANK / LOCAL_RANK / WORLD_SIZE
   ▼
setup_distributed()                     [train/fsdp_utils.py]
   ├─ torch.cuda.set_device(local_rank)  把本进程绑到对应 GPU
   └─ dist.init_process_group("nccl")    建立 NCCL 进程组
   ▼
模型 .to(cuda:local_rank)                每卡先放完整模型
   ▼
wrap_model_fsdp2(encoder) / (decoder)   [train/fsdp_utils.py]
   └─ fully_shard 把参数切成 DTensor，每卡只留 1/N
   ▼
优化器在【包装之后】创建                  关键：参数已是 DTensor，优化器状态也随之分片
   ▼
DataLoader（WebDataset split_by_node）   每卡读不同 tar shard（数据并行）
   ▼
训练循环（每个 step）:
   forward:  按层 all-gather 完整参数 → 算 → 丢弃碎片
   backward: 再次 all-gather → 算梯度 → reduce-scatter 把梯度分散归约回各卡
   step:     各卡用本地 1/N 梯度更新本地 1/N 参数
   ▼
保存 checkpoint:
   get_model_state_dict(full_state_dict=True)   汇聚完整权重到 rank 0
   get_optimizer_state_dict(full_state_dict=True) 汇聚完整优化器状态
   仅 rank 0 写盘 → checkpoint 与训练卡数解耦
   ▼
cleanup_distributed()                    销毁进程组
```

### 4.4 数据并行下，为什么各卡 loss 不同是正常的

这是一个容易误解的点，特意说明：

FSDP 属于**数据并行**——每张卡读的是**不同的数据分片**，所以各卡前向算出的 loss 本来就不一样。同步发生在**梯度层面**（backward 时的 reduce-scatter 把各卡梯度归约平均），不是 loss 层面。

所以**判断 FSDP 是否真正生效，不能看各卡 loss 是否相同**。正确的判据是日志里的诊断输出：

```
[FSDP 诊断] 参数已分片为 DTensor: True
[FSDP 诊断] 全量参数 4.87M | 本卡(rank 0)本地持有 1.29M (约 1/4)
```

`DTensor: True` + `本地持有 ≈ 全量/N` 才是真分片的铁证。这段诊断代码在
[train/train_stage1_ssl4eo.py](train/train_stage1_ssl4eo.py) 的 `main()` 中（FSDP 包装后）。

### 4.5 实测验证结果

| 配置 | 参数分片 | 每卡本地持有 | checkpoint |
|---|---|---|---|
| 单卡 | 不分片 | 4.87M（全量） | ✓ |
| 3 卡 FSDP | DTensor ✓ | 1.70M（≈1/3） | ✓ 干净 |
| 4 卡 FSDP | DTensor ✓ | 1.29M（≈1/4） | ✓ 干净 |

"checkpoint 干净"指：汇聚后不含任何 DTensor / DeviceMesh 引用，已实测可用单卡/CPU 加载 3 卡训练出的 checkpoint（卡数解耦）。

### 5.3 核心类与函数定义详解

**编码器** `TinyViTEncoder`（[models/encoders/tiny_vit_encoder.py](models/encoders/tiny_vit_encoder.py)）

```python
TinyViTEncoder(
    img_size=256,      # 输入边长
    in_channels=12,    # ← 模态相关：S2L2A=12, S2RGB=3
    patch_size=16,     # 每个 patch 16×16 → 256/16=16，共 16×16=256 个 patch
    embed_dim=256,     # token 维度
    depth=4,           # Transformer 层数
    num_heads=4,
)
# forward(x, mask_ratio=0.75) 返回 (latent, mask, ids_restore)
#   - 内部先 PatchEmbed 切 patch、加位置编码
#   - random_masking() 随机保留 25% patch，记录 mask 和还原索引
#   - 只把可见 patch 过 Transformer
```

**解码器** `LightDecoder`（[models/decoders/light_decoder.py](models/decoders/light_decoder.py)）

```python
LightDecoder(
    in_dim=256,            # = 编码器 embed_dim
    out_channels=12,       # ← 模态相关，须等于 in_channels
    patch_size=16,
    img_size=256,
    depth=2,
    decoder_embed_dim=128, # 解码器内部维度（默认 128，比编码器窄）
)
# forward(latent, ids_restore, mask) 返回重建图像 [B, C, H, W]
#   - 可见 token + 可学习 mask_token 拼回完整序列，按 ids_restore 还原顺序
#   - Transformer 解码 → 线性投影 → unpatchify 拼回图像
```

**FSDP 接入点**（[train/train_stage1_ssl4eo.py](train/train_stage1_ssl4eo.py) 的 `main()`）

```python
# 第 1 步：初始化分布式（torchrun 启动时才生效，否则单卡）
rank, local_rank, world_size, distributed = setup_distributed()
# 第 4 步：模型搬到本进程的 GPU
encoder = encoder.to(device); decoder = decoder.to(device)
# 第 5 步：仅多卡时用 FSDP2 包装
if distributed:
    encoder = wrap_model_fsdp2(encoder, mixed_precision="bf16")
    decoder = wrap_model_fsdp2(decoder, mixed_precision="bf16")
# 第 7 步：优化器必须在包装【之后】创建
optimizer = torch.optim.AdamW(params, ...)
```

## 六、现阶段做了什么、没做什么

### 6.1 已完成（Stage 1 范围内）

ObsWorld 的完整链条是：

```
历史遥感观测 + 成像条件 → 成像无关的地表状态
地表状态 + 外生驱动 + 地理先验 → 未来地表状态
未来地表状态 + 未来成像条件 → 未来遥感观测
```

**当前只做了第一环的预备件**：用 SSL4EO-S12 把"观测编码器（Observation Encoder）"通过 MAE 自监督预训练出来。具体完成项：

- ✅ **数据可读性验证**：确认 SSL4EO-S12-v1.1 真实可流式读取，摸清 7 模态 / 477 shard / zarr 结构
- ✅ **数据流水线**：WebDataset + zarr.zip 解析，输出统一字典（image + phi + field_mask）
- ✅ **最小模型**：TinyViT 编码器 + 轻量解码器 + masked 重建损失（4.87M）
- ✅ **单卡训练闭环**：forward → loss → backward → optimizer → checkpoint 全通
- ✅ **多卡 FSDP 真分片**：1/2/3/4 卡实测，参数 DTensor 化、每卡持有 1/N
- ✅ **checkpoint 卡数解耦**：汇聚完整权重，3 卡训练的可在单卡/CPU 加载

### 6.2 已埋好的扩展接口（为后续阶段预留）

数据返回的字典里，已经预留了 ObsWorld 后续需要的字段，当前不参与训练但接口在：

```python
{
  'image': ...,          # 当前训练用
  'phi': {               # 成像条件（Imaging Condition）—— Stage 2/3 要用
    'sensor', 'season', 'cloud_mask', 'lat', 'lon', 'time'
  },
  'modality': ...,       # 模态 id —— 多模态融合用
  'field_mask': {...},   # 字段有效性掩码 —— 处理字段缺失用
}
```

时序维度也保留了：`random_season=false` 时数据返回 `[4季, C, H, W]`，Stage 2 的状态动力学可直接用。

### 6.3 明确没做（留给 Stage 2/3/4）

- ❌ State Dynamics Module（状态动力学，外生驱动 D + 地理先验 G）
- ❌ 未来地表状态预测
- ❌ Conditional Observation Decoder（未来成像条件解码未来观测）
- ❌ 扩散模型 decoder
- ❌ 下游任务头（洪水 / 土地覆盖 / 建筑变化）
- ❌ 大规模正式训练（当前是 tiny 模型 + 10 步 smoke）

### 7.6 注意事项

- **学习率写法**：YAML 里写 `0.0001` 而非 `1e-4`——PyYAML 会把 `1e-4` 解析成字符串导致报错（这是之前踩过的坑，已在配置注释里标注）。
- **num_workers**：当前配置设为 0（WebDataset + 多卡下最稳）。正式训练且单机数据吞吐成瓶颈时可调大。
- **切模态后两处都要改**：`data.modality` 和 `encoder.in_channels`/`decoder.out_channels` 必须同步。
