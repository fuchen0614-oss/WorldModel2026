## 1. 执行摘要

> [!important]
> **核心结论:S1 的升/降轨(asc/desc)可以 100% 可靠、纯离线地补入 phi;入射角(incidence angle)廉价不可行,建议先占位。**

来龙去脉:SSL4EO 的官方 metadata parquet 里**没有**任何轨道/入射角/产品 ID 字段,但 **S1GRD 的 zarr 内部藏着一个 `file_id` 数组,存的是完整的原始 Sentinel-1 产品 ID**(每个时间片一个)。从产品 ID 可以离线解析出卫星、模式、极化、绝对轨道号等,再由"绝对轨道号 + 成像时刻 + 经度"用太阳同步轨道局地时(LST)法推算升/降轨。

我用 Microsoft Planetary Computer 的 STAC 精确 `sat:orbit_state` 做了真值验证:**离线推算与真值在 38 个可验证样本上一致率 100%(含升轨与降轨两类)**,相对轨道公式也与 STAC 完全吻合。

## 2. 本地字段审查结果(任务 A)

### 2.1 官方 metadata 与 tar/key 命名:无几何字段

### 2.2 🎯 关键发现:S1GRD zarr 的 `file_id` = 完整原始产品 ID

S1GRD 的 zarr 内含数组 `file_id: shape=(4,) dtype=<U67`,每个时间片一个完整 Sentinel-1 产品名:

```
S1A_IW_GRDH_1SDV_20200307T145157_20200307T145222_031571_03A32A_72A2
└┬┘ └┬┘ └─┬─┘ └┬─┘ └──────┬─────┘ └──────┬─────┘ └──┬─┘ └──┬─┘ └┬┘
卫星 模式 产品   极化      起始时间          结束时间       绝对轨道 datatake 唯一码
```

| 片段 | 含义 | 用途 |
|---|---|---|
| `S1A`/`S1B` | 卫星 | phi 候选(轻量) |
| `IW` | 干涉宽幅模式 | 一致性核查(全 IW) |
| `GRDH` | Ground Range Detected High-res | 一致性核查 |
| `1SDV` | L1 / 双极化 VV+VH | 一致性核查(全 DV) |
| `20200307T145157` | 成像起始 UTC | 推 asc/desc(局地时) |
| `031571` | **绝对轨道号** | 推相对轨道 + 验证 asc/desc |
| `03A32A` / `72A2` | datatake / 唯一码 | STAC 匹配用 |

> [!note]
> `build_phi_cache.py` 的注释里其实早已记录 `file_id: [4] str (文件ID)` 是"真实 zarr 字段(已验证)",只是从未被解析进 phi。这意味着现有离线 pipeline 已经在读这些 zarr,补 file_id 解析几乎零额外 I/O 成本。

### 3.2 asc/desc:STAC 精确真值验证离线推算

由于 zarr 有完整产品 ID,可用"成像时间窗 ± 20min + 中心点 bbox"在 PC STAC `sentinel-1-grd` 集合按 `sat:absolute_orbit` 精确匹配,取 `sat:orbit_state` 作真值。

> [!important]
> **验证结果:抽 40 个产品(刻意覆盖升/降轨两类),38 个在 STAC 命中,离线 LST 推算与真值 `sat:orbit_state` 一致 38/38 = 100%。** 另 2 个 STAC 时间窗内无匹配(abs_orbit 37302、20932),记为"未验证"而非"不一致"。
> 相对轨道公式同步验证:S1A abs 31571 → 174、S1B abs 21900 → 174,均与 STAC `sat:relative_orbit` 完全吻合。

离线推算法(`orbit_direction_from_lst`):Sentinel-1 升交点平局地时(MLTAN)= 18:00,故升轨整条 pass 局地太阳时 ≈ 18:00(区间 [12,24)),降轨 ≈ 06:00(区间 [0,12));`LST = UTC小时 + 经度/15`。**因为 S1 是太阳同步轨道,此法对中低纬高度可靠;高纬极区有边界歧义(SSL4EO 中占比低)。** 脚本:`scripts/validate_orbit_direction.py`;结果:`outputs/data_inspection/orbit_direction_validation.json`。

> [!note]
> 由于一致率 100% 且离线推算无需联网,**phi v3 全量构建(246k 样本)推荐纯离线**(LST + 相对轨道公式),不依赖 STAC。STAC 仅用于本次抽样验证。这避免了 ~百万次网络查询的耗时与失败风险。

### 3.3 incidence angle:STAC 不暴露,annotation 路线昂贵

> [!warning]
> **PC STAC 场景级属性不含入射角。** 实际 `sar:` 属性只有 looks/resolution/pixel_spacing/polarizations/observation_direction 等;`observation_direction` 对 S1 恒为右视,无区分度。`有无 incidence/angle 字段: []`。

要拿真实入射角只能走原始产品 annotation XML 的 geolocationGrid `incidenceAngle`。代价与风险:
- Sentinel-1 IW 入射角在 250km 幅宽内约 **29°–46°** 渐变;而 SSL4EO patch 仅 264×264@10m ≈ 2.6km,**在幅宽内的横向位置未知**,无法仅凭产品 ID 定位。
- 要精确:对每个唯一产品(90 样本即含 83 个唯一相对轨道,全量唯一产品可能数万至十万级)下载 annotation XML,再用 patch 的地理 footprint 在 geolocation grid 上插值——大批量下载,且老产品可能移出热存储、有 API 限流与失败风险。
- 退而求场景级平均(~38°):几乎对所有样本恒定,**作为 phi 几乎无判别信号**,不值得。

**结论:incidence angle 当前不纳入 phi 主训练。** 保留 schema 占位字段 `s1_incidence_angle` + `field_mask=0`,在报告"下一步"列出未来可行路线。
