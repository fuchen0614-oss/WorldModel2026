# 周报（2026-W26，截至 2026-06-28）

项目：ObsWorld　阶段：Stage 1 → 1.5　记录人：Zhijian Liu

本周主要做了四件事：将 Stage 1 从 tiny 验证版升级为正规规模模型（ViT-S/16, 22.77M）并开始训练；把 phi 成像条件字段全部调试好；在 tiny 版上跑通了 Stage 1.5 成像解耦全链路；并用 EuroSAT 对现有结果做了精度标定。

---

## 背景

ObsWorld 要做的不是“预测下一帧卫星图”，而是先从历史遥感观测里学出成像无关的地表状态，再预测它的未来演化，最后解码成指定成像条件下的未来观测。整条路线分 5+1 个阶段，本周在最前两个阶段，涉及几个名词：

- Stage 1（观测编码器预训练）：用 MAE（遮住 75% 图像块让模型重建）在 SSL4EO-S12 上预训练编码器，把 Sentinel-1 雷达 + Sentinel-2 光学双模态影像压成特征，是整个框架的地基。
- phi（成像条件 φ）：同一块地，太阳高度角、季节、云量不同，拍出来就完全不同。这些只跟“怎么拍”有关、与地表无关的因素统称 phi，抽出来后模型才能剥离成像差异、只留地表状态。
- Stage 1.5（成像解耦）：在 Stage 1 上把 phi 经 FiLM 注入编码器，加一个解耦损失，让特征不随成像条件变化，是 Stage 1 到 Stage 2 的过渡。

---

## 一、Stage 1：tiny 验证版 → 正规规模并开训

此前用 tiny 验证版（encoder 5.7M，embed=256/depth=6）主要是为了先打通双模态 MAE 流程、排掉工程坑（数据加载、多卡、phi join）。流程确认无误后，本周换成对标论文的正规 ViT-S/16（22.77M，embed=384/depth=12），开始正式预训练，配置已完全对齐 SSL4EO-S12 论文的 MAE ViT-S/16，可直接横向比较。训练步数 50k → 95k（约 200 patch-epoch）。

开训时碰到一个性能问题：GPU 利用率在 0% 和 100% 之间剧烈抖动、step time 高达 7.5s。诊断是数据所在网络盘读带宽饱和、喂不上 GPU（不是算力或 batch 问题）。解法是把数据从网络盘复制到本地高速介质——S2L2A（842G）放运存、S1GRD（168G）放本地 SSD，软链拼出原目录结构、训练代码零改动，原始数据集不动、跑完清理副本即可。优化后单样本读取 237ms → 4ms，step time 7.5s → 1.97s，GPU 稳定满载，总时长从约 8.3 天压到约 2.6 天。

排查时还发现一个问题：训练脚本的 checkpoint_interval 只读命令行、没读配置文件，导致中间权重不落盘、只在终点存一次。本次只需最终权重，已确认继续训练，问题记下次开训前修。

## 二、phi 成像条件字段全部调试好

把 phi 从原始 zarr 离线抽取、清洗、缓存成 parquet（一次处理、永久复用，便于离线验证）。当前覆盖全量 24.4 万样本、964 个 parquet、46 个字段（从 v1.0 的 22 增至 46），存于 `phi_processed/`。

**完整 phi 字段表（46 个）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| sample_key | 标识 | 样本唯一键（tar __key__，用于 join） |
| modality / sensor / product_level | 元信息 | S1/S2、传感器型号、产品级别 |
| center_lat / center_lon / crs | 地理 | 中心纬经度、坐标系 |
| time_0 ~ time_3 | 时间 | 4 个时间片的时间戳（纳秒） |
| time_valid_0 ~ time_valid_3 | 时间 | 各时间片是否有效 |
| season_0 ~ season_3 | **成像条件** | **季节（0-3 对应春夏秋冬，v2.0 新增）** |
| day_of_year_0 ~ day_of_year_3 | **成像条件** | **一年中第几天（v2.0 新增）** |
| sun_elevation_0 ~ sun_elevation_3 | **成像条件** | **太阳高度角（v2.0 新增，NOAA 公式）** |
| cloud_cover_0 ~ cloud_cover_3 | **成像条件** | **云量（v2.0 修复误判 bug）** |
| cloud_shadow_0 ~ cloud_shadow_3 | 质量 | 云影比例（v2.0 新增） |
| valid_ratio_0 ~ valid_ratio_3 | 质量 | 有效像素比例（v2.0 新增） |
| num_timesteps / num_bands / height / width / spatial_resolution | 数据属性 | 时间片数、波段数、尺寸、分辨率 |
| sample_id / _field_mask | 内部 | 样本 ID、字段掩码 |

**本周关键修复：**
- 云量算错（bug）：v1.0 把 water/snow/云影/no_data 全算成云，v2.0 改为只算 thin+thick cloud。实测同一水域样本云量从误报的 94.1% 修正到 13.0%（67% 水体曾被误判为云）。
- 补 sun_elevation 和 season——成像解耦最强的两个干扰因子，FiLM 可直接用。
- view_angle（观测角）因源数据缺失暂搁置，S2 视场窄影响有限。

## 三、Stage 1.5 成像解耦链路（tiny 版）跑通

Stage 1 编码器只吃图像、没用 phi。Stage 1.5 把 phi 经 FiLM 注入（X + φ → 成像无关状态 s_t），加解耦损失约束“换成像条件、特征不变”。本周三个新模块加训练逻辑（ImagingConditionEncoder、FiLM 调制、ImagingDecouplingLoss、季节对比 loop）全部写完，6/6 集成测试通过——关键验证了 FiLM identity（phi=0 时与不注入完全一致，差异 0）和能正确加载 Stage 1 权重。

调试中解决两个坑：phi join 主键必须用 tar 的 __key__（不是 zarr 的 sample 字段），以及 S1 无云字段导致前向出 NaN（需对缺失值做保护）。修复后链路可稳定训练，等正规版 Stage 1 完成即可加载权重正式开训。

## 四、EuroSAT 精度标定（调试期顺带做的）

调试 Stage 1 编码器质量时，顺带在 EuroSAT（27,000 张 S2 图像，10 类）做了 Linear Probing（冻结 encoder 只训线性头）。注意：这是调试期 tiny 版（5.7M, 50k steps）的结果，不是本周开训的正规版，正规版预计显著更高。

**与现有方法对比（均在 SSL4EO-S12 上预训练、单模 S2、linear probing）：**

| 方法 | Backbone | 参数量 | EuroSAT OA |
|------|----------|--------|-----------|
| MAE（对标） | ViT-S/16 | 22M | 94.1% |
| DINO | ViT-S/16 | 22M | 97.7% |
| MoCo v3 | ResNet-50 | 26M | 98.0% |
| 我们·单模 S2 | tiny ViT | 5.72M | 70.43% |
| 我们·双模 S1+S2 | tiny ViT | 5.72M | 69.57% |

偏低主要是参数量和训练量的问题，不是方法问题。我们调试版只有 5.72M（基线 22M 的 26%）、depth 6（基线 12）、训练 50k（双模轮流 S2 实际仅约 25k）。按因素拆解：模型规模 −10~15%、双模态稀释 −5~8%、训练长度 −3~5%、架构细节 −2~4%，合计约 −20~32%，与实际 −24.5% 吻合。且 per-class 精度从 SeaLake 99% 到 Highway 27% 呈合理梯度（非随机），说明确实学到了表征——粗粒度类（水体/森林）很好，细粒度类（公路/细分作物）受限于容量。是一个可解释、可接受的 baseline。

本周开训的正规版把容量补到 22M、步数翻倍到 95k，预期精度区间 80~92%、向 94.1% 靠拢——这次正规化正是为了补上这两块短板、验证框架在标准规模下的真实上限。
