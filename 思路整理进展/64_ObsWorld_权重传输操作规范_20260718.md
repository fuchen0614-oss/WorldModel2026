# 64 ObsWorld：权重传输操作规范（GitHub Releases）

> 日期：2026-07-18
> 性质：操作规范文档。用于在**能连 GitHub、但连不上 Hugging Face**的服务器上，把模型权重（checkpoints）传出去 / 在别的机器下回来。
> 通道：GitHub **Release 附件**（每个文件上限 2GB、不占仓库体积、不吃 LFS 配额、免费）。
> 关联：本文件另有一份同步副本在 `scripts/权重传输操作规范.md`。
> 适用现状：本项目 checkpoint 都是 130–350MB 的小 `.pt`，**远低于 2GB，直接传，无需切块**。

---

## 0. 一次性准备（只做一次）

1. **装 gh（免 root，从 github.com 下二进制）**
   ```bash
   GH_VER=$(curl -s https://api.github.com/repos/cli/cli/releases/latest | grep -oP '"tag_name":\s*"v\K[^"]+')
   curl -L "https://github.com/cli/cli/releases/download/v${GH_VER}/gh_${GH_VER}_linux_amd64.tar.gz" -o /tmp/gh.tgz
   tar -xzf /tmp/gh.tgz -C "$HOME"
   echo "export PATH=\"\$HOME/gh_${GH_VER}_linux_amd64/bin:\$PATH\"" >> ~/.bashrc
   export PATH="$HOME/gh_${GH_VER}_linux_amd64/bin:$PATH"
   gh --version
   ```

2. **建一个空的 private 仓库**（网页操作，不勾 README）：`WorldModel2026-weights`

3. **造一个带权限的 Token（关键，别再用 scopes=none 的）**
   - classic：Settings → Developer settings → Tokens (classic) → 勾 **`repo`**。
   - 或 fine-grained：对该仓库给 **Contents: Read and write**。
   ```bash
   export GH_TOKEN=ghp_你的token          # 建议也写进 ~/.bashrc
   gh auth status                          # scopes 里应能看到 repo（不是 none）
   ```
   > ⚠️ Token 是密码级机密：不要贴进聊天/日志/commit；一旦外泄立即到 Tokens 页面 Revoke。

## 1. 使用规范（优先看这里）

### 1.1 上传：一次传三个阶段（stage1 / 1.5 / 2）

> 每个阶段一个独立 release（避免不同目录里同名 `checkpoint_best.pt` 撞名）。
> **新开的 shell 记得先重设 `GH_TOKEN` 和 gh 的 `PATH`（见 §0），它们不会跨 shell 保留。**

```bash
REPO=fuchen0614-oss/WorldModel2026-weights
BASE=/csy-mix02/cog8/zjliu17/Agent/WorldModel2026/checkpoints

declare -A STAGES=(
  [stage1]="stage1_vits_dual_staged"
  [stage1.5]="stage1_5_dual_conditioned_vits_state_bridge_60k"
  [stage2]="stage2_physical4_8gpu_b64_200ep_trainvalcache_20260717_165126"
)

for TAG in stage1 stage1.5 stage2; do
  DIR="$BASE/${STAGES[$TAG]}"
  echo "==================== $TAG : $DIR ===================="
  [ -d "$DIR" ] || { echo "!! 目录不存在,跳过"; continue; }
  cd "$DIR"
  ls *.pt >/dev/null 2>&1 || { echo "!! 没有 .pt,跳过"; continue; }
  echo "将上传:"; ls -lh *.pt
  # 建 release（只在不存在时建；这次不吞错误，好看到真正的失败原因）
  gh release view "$TAG" --repo "$REPO" >/dev/null 2>&1 \
    || gh release create "$TAG" --repo "$REPO" --title "$TAG" --notes "$TAG checkpoints"
  sha256sum *.pt > /tmp/SHA256SUMS.txt
  gh release upload "$TAG" --repo "$REPO" --clobber *.pt /tmp/SHA256SUMS.txt
  echo "---- $TAG 附件 ----"; gh release view "$TAG" --repo "$REPO"
done
echo "全部完成: https://github.com/$REPO/releases"
```

- **只想传里程碑**（best + epoch 检查点，不含中间 step 快照）：把两处 `*.pt` 换成
  `checkpoint_best.pt checkpoint_epoch*_step_*.pt`
- **只传某一个阶段**：把 `for TAG in stage1 stage1.5 stage2` 改成 `for TAG in stage2`。

### 1.2 下载：在另一台机器还原

```bash
REPO=fuchen0614-oss/WorldModel2026-weights   # 需先装 gh + export GH_TOKEN
for TAG in stage1 stage1.5 stage2; do
  gh release download "$TAG" --repo "$REPO" --dir "./restored/$TAG"
  ( cd "./restored/$TAG" && sha256sum -c SHA256SUMS.txt )   # 全 OK = 字节一致
done
```

### 1.3 成功的标志
`gh release view` 会列出该 release 的附件清单（N 个 `.pt` + 1 个 `SHA256SUMS.txt`），
且下载端 `sha256sum -c` 全部 `OK`。看到 `release not found` / `HTTP 403` 就是**没成功**。

## 2. 常见报错对照

| 现象 | 原因 | 处理 |
|---|---|---|
| `release not found` | `gh release create` 没建成（多半 token 无权限），后续 upload 找不到 release | 换**带 `repo` 权限**的 token（§0.3） |
| `HTTP 403` / `Resource not accessible by personal access token` | token 权限不足 | 同上 |
| `Token scopes: none` | 用了没勾任何 scope 的 classic token | 重建并勾 `repo` |
| `No such file or directory` / `ls` 列出的是别的目录 | **新开 shell**，之前的 `cd`/变量丢了 | 用**绝对路径** `cd`，重设 `GH_TOKEN`/`PATH` |
| `Command 'gh' not found` | 新 shell 没加载 gh 的 PATH | `export PATH="$HOME/gh_*/bin:$PATH"`（§0.1 已写进 .bashrc） |
| 上传报文件 >2GB | 单个附件超过 Release 上限 | 先切块（§3.3） |

## 3. 脚本是哪个 / 原理（其次了解）

### 3.1 核心原理
- 普通 `git push` 对单文件有 **100MB 硬限制**（超过直接拒收）。
- 但 **GitHub Release 的附件**每个可到 **2GB**，且**不计入仓库体积、不占 Git LFS 配额、免费**。
- 本项目 checkpoint 都 <2GB → **直接当 Release 附件传，不用切块**，这就是 §1 的做法。
- `SHA256SUMS.txt` 记录每个文件的哈希，下载端 `sha256sum -c` 保证两边**逐字节一致**。

### 3.2 现成脚本（封装版，适合“单个大文件 / 整个目录打成一个包”）
仓库 `scripts/` 下有两个脚本，会自动 **打包→切块(<2GB)→sha256→Release 上传/下载重组**：
- `scripts/upload_weights_via_github.sh <文件或目录> <tag> [owner/repo]`
- `scripts/download_weights_via_github.sh <tag> <out-dir> [owner/repo]`

它们适合“把某个 >2GB 的大文件、或把整个目录当**一个**归档”上传。
**对本项目这种“很多个 <2GB 的小 .pt”，用 §1 的内联命令更好**（每个 `.pt` 直接是一个可单独下载的附件，不必先 tar 成一个大包）。

### 3.3 单文件 >2GB 时的切块（备用）
```bash
# 上传端：
split -b 1900m -d --suffix-length=3 大文件.pt 大文件.pt.part-
sha256sum 大文件.pt.part-* > /tmp/SHA256SUMS.txt
gh release upload "$TAG" --repo "$REPO" --clobber 大文件.pt.part-* /tmp/SHA256SUMS.txt
# 下载端：
gh release download "$TAG" --repo "$REPO" --dir ./restored
cd ./restored && sha256sum -c SHA256SUMS.txt && cat 大文件.pt.part-* > 大文件.pt && rm 大文件.pt.part-*
```

## 4. 一句话流程
新 shell → 重设 `GH_TOKEN`(带 repo 权限) + gh 的 `PATH` → 跑 §1.1 上传 → 另一台跑 §1.2 下载校验。
`release not found` / `403` 永远先查 token 权限。
