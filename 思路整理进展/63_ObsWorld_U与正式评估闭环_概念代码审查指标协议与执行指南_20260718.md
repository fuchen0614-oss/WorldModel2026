## 0. 一页结论：现在到底做到哪里

### 0.1 对当前训练的判断

如果已经完成的 200 epoch Stage2 使用的是：

```text
configs/train/stage2_earthnet_v2_direct_physical4.yaml
```

那么完成的是 **Direct-P4**，不是 Rollout-P4，也不是 U。

这里四个词不要混在一起：

| 名称 | 它表示什么 | 是否等于另一个概念 |
|---|---|---|
| Stage2 | 当前长期 EarthNet 预测训练阶段 | 不等于 Rollout |
| P4 / physical4 | 使用 4 个物理天气驱动 | 不等于“第 4 阶段” |
| Direct-P4 | physical4 驱动下的非递推直接预测 | 不等于 Rollout-P4 |
| Rollout-P4 | physical4 驱动下反复复用共享五日转移 | 需要另一条独立训练 |
| U | 重新得到未来观测后更新内部状态 | 必须建立在可用 Rollout 上 |

因此，完成 Direct-P4 后的正确顺序是：

```text
Direct-P4 已训练
  → 在完整 val_dev 上选择 Direct 的一个 checkpoint
  → 冻结 Direct 结果
  → 单独训练 Rollout-P4
  → 在完整 val_dev 上选择 Rollout 的一个 checkpoint
  → Direct 与 Rollout 做同协议正式比较
  → 只有 Rollout 通过后，才初始化并训练 U
```

### 0.2 对评估代码的最新判断

本轮发现工作区有三处尚未提交的评估修改：

```text
eval/eval_stage2_earthnet.py
eval/predict_stage2_earthnet.py
eval/eval_observation_correction.py
```

三处都补上了：

```python
batch = prepare_stage2_batch_for_model(batch, data_cfg)
```

这个修改是**正确且必要的**。它让评估与训练一样，在 `defer_context_resize_to_device=true` 时把 context 从原生 128×128 恢复为 Stage1.5 encoder 需要的 256×256。

但它只修好了：

1. Stage2 内部评估；
2. legacy EarthNet NPZ 预测导出；
3. 当前 U 评估入口。

它**尚未修复** `eval/export_greenearthnet_predictions.py`。因此，准备与 Contextformer 论文公开 R²/RMSE 等数值同协议比较时，当前 NetCDF 导出链仍不能直接作为正式结果。

### 0.3 当前存在两套不同的“正式指标”

| 评估协议 | 输出 | 指标 | 当前用途 |
|---|---|---|---|
| legacy EarthNet2021 / EarthNetScore | NPZ | ENS、MAD、OLS、EMD、SSIM | EarthNet legacy 补充结果 |
| GreenEarthNet CVPR 2024 | NetCDF NDVI | R²、RMSE、NSE、绝对 bias、Outperformance、RMSE25 | 与 Contextformer/PredRNN/SimVP 论文值比较 |

这两套指标不是同一套 evaluator（评估器），不能因为都叫 EarthNet 就混为一谈。

### 0.4 对 U 的最新判断

U 的模型、因果顺序、训练接线和基础 evaluator 已经存在；但论文正式 U 结果仍缺：

1. 从**选定且冻结的 Rollout-P4 checkpoint**初始化 U 的安全加载路径；
2. 冻结 I/F/O，只训练 U，或明确受控解冻策略；
3. 固定 day-25/day-50 reveal，而不是随机 reveal 的正式评估；
4. 每样本配对 Gain、Gain-AUC 和 no-reveal 严格等价检查；
5. observation mask（重新观测有效区域）的最终协议；
6. VanillaFilter 的独立训练和容量匹配；
7. 完整 provenance（来源记录）与统计检验。

因此当前 U 是：

> **核心代码已接通，但论文证据闭环尚未完成。**

## 1. Persistence 到底是什么

### 1.1 最简单的解释

Persistence（持续性基线）就是：

> 假设未来不会变化，把历史中最后一次有效观测一直复制到未来。

例如最后一次看到的 NDVI 是 0.62，那么它预测：

```text
day 5   = 0.62
day 10  = 0.62
day 15  = 0.62
...
day 100 = 0.62
```

它没有神经网络、没有参数，也没有 epoch：

```text
Persistence 的训练 epoch = 0
```

它存在的意义不是和 ObsWorld 比模型复杂度，而是回答：

> 我们训练了这么大的模型，是否真的比“什么都不变”更好？

### 1.2 当前仓库中的 Persistence

`eval/forecast_metrics.py` 已经有一个内部 Persistence-MAE：

1. 对每个像元寻找历史 context 中最后一次 clear observation（清晰有效观测）；
2. 将该像元的最后有效 RGBN 值复制到所有未来时刻；
3. 使用当前 `target_mask` 计算 MAE；
4. 输出 `persistence_MAE` 和 `skill_vs_persistence`。

但这只是**当前 Stage2 内部监控基线**，它不自动等于 Contextformer 论文 Table 2 的 Persistence 数值。

原因是 Contextformer 使用的是另一套完整协议：

- 预测目标是 NDVI；
- clear mask 使用 `s2_dlmask` 和 SCL 类别；
- 只保留满足植被和观测质量条件的像元；
- 按 minicube 和 land-cover（地类）进行平衡聚合；
- R²/RMSE/NSE/bias 的计算和聚合顺序固定。

所以“都是 Persistence”不代表“分数可直接互换”。

### 1.3 能不能直接使用别人论文中的 Persistence 数值

可以，但分两种口径。

#### 口径 A：严格直接比较

如果我们确认以下内容与 Contextformer 完全一致：

1. 官方数据 release（发布版本）；
2. 官方测试文件集合；
3. context/target 时间范围；
4. clear/SCL mask；
5. high-quality pixel（高质量像元）筛选；
6. NDVI 公式；
7. minicube/land-cover 聚合；
8. evaluator commit（评估器提交版本）；

那么可以直接引用论文中的 Persistence、Climatology、PredRNN、SimVP、Contextformer 数值，不需要重新训练这些模型。

这时真正需要“跑一遍”的是：

> 用完全相同的官方 evaluator 评估我们的 Direct-P4 和 Rollout-P4。

#### 口径 B：只作为公开参考值

如果我们的 frozen manifest、训练划分、测试文件或 mask 与论文不完全一致，则可以引用，但必须标注：

```text
† Published result under the original GreenEarthNet protocol;
  not rerun under our frozen manifest.
```

中文意思：

> † 为原论文协议下的公开结果，未在本文冻结 manifest 下重跑，仅用于量级参考。

此时不能写“我们的结果严格超过 Contextformer，因此达到 SOTA”。

### 1.4 为什么建议自己再算 Persistence/Climatology

不是因为它们要训练，而是因为成本很低、证据价值很高：

- 不需要 GPU 训练；
- 不需要 200 epoch；
- 可以确保使用和我们完全相同的测试样本、mask 与聚合；
- Outperformance 本身需要逐像元 Climatology 对照分数，不能只靠论文中的一个汇总数字恢复。

因此推荐：

> 学习型外部模型可以优先使用论文公开值；Persistence 和 Climatology 最好在本文测试协议上自己生成一次。

## 2. “我训练了 200 epoch”，为什么还要选 checkpoint

### 2.1 checkpoint 是什么

训练 200 epoch 的过程中会保存多个模型快照，例如：

```text
checkpoint_best.pt
checkpoint_epoch100_step_4400.pt
checkpoint_epoch150_step_6600.pt
checkpoint_epoch200_step_8800.pt
checkpoint_step_8800.pt
```

它们都是同一条训练中的不同时间点。

200 epoch 的最后一个模型不一定最好。例如：

- epoch 100：val MAE = 0.030；
- epoch 150：val MAE = 0.027；
- epoch 200：val MAE = 0.029。

那么应该选 epoch 150，而不是因为“训练计划写了 200 epoch”就强制选最后一个。

### 2.2 checkpoint 选择不是重新训练

checkpoint selection（checkpoint 选择）只做：

```text
读取已经保存的候选 checkpoint
  → 在完整 val_dev 上逐个预测
  → 用预先写死的指标排序
  → 选择一个
  → 写 selected_checkpoint.json
```

它不会：

- 更新模型参数；
- 重新训练 200 epoch；
- 使用 IID/OOD test 选模型。

可以把它类比成：

> 200 epoch 产生了几位候选选手；val_dev 是模拟考试；IID/OOD 是最终考试。只能用模拟考试选选手，不能偷看最终考试答案。

### 2.3 为什么不能直接使用 checkpoint_best.pt

当前训练配置中的 `checkpoint_best.pt` 是按每 1000 steps、固定最多 512 个 val 样本监控选出的。它适合判断训练有没有坏掉，但不等于在完整 `val_dev` 上最优。

当前 `eval/select_stage2_checkpoint.py` 已经专门解决这个问题：

- 默认发现 `checkpoint_best.pt`；
- 发现 epoch100/150/200 的命名 checkpoint；
- 对每个候选跑完整指定 split；
- 默认按 `MAE` 最小选择；
- 输出每个候选的指标和最终选择。

### 2.4 你现在到底完成了 Direct 还是 Rollout

用 checkpoint 中保存的 resolved config（完整合并配置）确认，不要靠文件夹名字猜：

```bash
cd /csy-mix02/cog8/zjliu17/Agent/WorldModel2026
source /csy-opt/cog8/zjliu17/miniconda3/bin/activate WorldModel

CHECKPOINT=/csy-mix02/cog8/zjliu17/Agent/WorldModel2026/checkpoints/stage2_earthnet_v2_direct_physical4/checkpoint_epoch200_step_8800.pt

python -c 'import sys,torch; d=torch.load(sys.argv[1],map_location="cpu",weights_only=False); c=d["config"]; print("forecast_mode =",c["model"]["forecast_mode"]); print("driver_protocol =",c["data"]["driver_protocol"]); print("global_step =",d.get("global_step"))' "$CHECKPOINT"
```

判断方法：

| 输出 | 含义 |
|---|---|
| `direct_path_physical4` | 已完成 Direct-P4 |
| `rollout_t5_physical4` | 已完成 Rollout-P4 |
| `rollout_t5_physical4_correction` | U/Filter 模型 |

根据当前代码和执行文档，已经完成/正在完成的 200 epoch 主训练候选是 **Direct-P4**；Rollout-P4 使用另一份配置 `stage2_earthnet_v2_rollout_physical4_h200_200ep.yaml`，必须单独训练。

## 3. epoch 是否必须和别人一样

### 3.1 外部公开 benchmark 对比

不要求所有方法都训练 200 epoch。

公开 benchmark 中，不同论文通常使用各自报告的训练方案。公平性的核心是：

- 合法且相同的数据任务；
- 相同测试集；
- 相同输入/输出定义；
- 相同 evaluator；
- 不使用 test 调参；
- 报告训练设置和模型规模。

所以：

> 我们训练 200 epoch，不会让别人论文中的 100 epoch 或其他训练计划自动失效。

### 3.2 项目内部因果式对照

但 Direct-P4 与 Rollout-P4 的比较必须尽量匹配：

- 同一 train/val manifest；
- 同一 Stage1.5 初始化；
- 同一 physical4；
- 同一全局 batch；
- 同一 8,800 optimizer steps / 200 epoch 预算；
- 同一 loss、解码器、随机种子；
- 唯一关键区别是 Direct 与共享递推。

因为这对比较要回答的不是“谁调参更多”，而是：

> 共享短步状态递推本身是否有价值？

### 4.1 58 文档当前拟定的正文指标

正文拟定了：

### 4.2 当前代码覆盖矩阵

| 指标 | 当前代码状态 | 是否需要重训模型 | 是否需要重跑推理/评分 |
|---|---|---:|---:|
| MAE | 已实现 | 否 | 若只有 checkpoint，需要评估 |
| NDVI-MAE | 已实现 | 否 | 若只有 checkpoint，需要评估 |
| 5–100 天每 5 日 MAE/NDVI-MAE | 已实现 | 否 | 需要评估 |
| long-horizon MAE | 已实现 | 否 | 需要评估 |
| skill vs Persistence | 已实现（内部 MAE 口径） | 否 | 需要评估 |
| ENS/MAD/OLS/EMD/SSIM | legacy EarthNet 路径已实现 | 否 | 需要 NPZ 导出和评分 |
| R²/RMSE/NSE/绝对 bias/RMSE25 | `greenearthnet_protocol.py` 的公式和聚合已实现 | 否 | 需要修好 NetCDF 导出后评分 |
| Outperformance | 汇总公式已实现 | 否 | 需要同像元 Climatology score；当前 baseline 生成链缺失 |
| 5–100 天逐点 R²/RMSE 曲线 | 尚未实现；现有 GreenEarthNet 仅四个 25 天区间 | 否 | 需补 metric code，再评分现有预测 |
| tile-cluster bootstrap 95% CI | 尚未实现 | 否 | 若保存逐样本结果，可直接 bootstrap |
| 3 seed mean±std | 结果未具备 | **是，需要独立训练 seed** | 是 |
| Params/FLOPs | 尚未形成正式 bundle | 否 | 加载模型统计即可 |
| latency/throughput | 训练日志有部分吞吐；正式推理 benchmark 未闭环 | 否 | 需要单独 benchmark |
| U 的 Gain-AUC/固定 reveal 指标 | 尚未实现 | U checkpoint 需要训练 | 需要专门评估 |

### 4.3 “重跑”要区分三种事情

#### A. 重新训练

只有这些情况需要重新训练：

- Rollout-P4 还没有训练；
- U 还没有从冻结 Rollout 正确初始化并训练；
- VanillaFilter 需要独立训练；
- 论文坚持报告 3 seed，而目前只有 1 seed；
- Stage1 vs Stage1.5 等真正改变模型的消融。

#### B. 重新跑模型推理

如果只有 checkpoint、没有保存完整预测文件，则计算新指标需要重新跑一遍 inference（推理）。这不是训练，通常快得多。

#### C. 只重新评分

如果已经保存了完整、带 manifest/hash 的逐样本预测，那么新增 R²/RMSE/CI 等通常只需重新跑 scorer（评分器），不需要再次经过 GPU 模型。

所以最省时间的正式策略是：

> 每个最终 checkpoint 只导出一次完整预测，并永久保存；所有指标、表格和 bootstrap 都从冻结预测文件重新计算。

## 5. 两套 evaluator 必须严格区分

### 5.1 legacy EarthNetScore

代码链：

```text
eval/predict_stage2_earthnet.py
  → 20 帧 RGBN NPZ
  → eval/score_earthnet_prediction_dir.py
  → EarthNetScore.get_ENS
```

主要输出：

- ENS；
- MAD；
- OLS；
- EMD；
- SSIM。

它适合作为 EarthNet legacy 补充指标，但不能直接产生 Contextformer Table 2 的 R²/RMSE/NSE。

### 5.2 GreenEarthNet CVPR 2024 evaluator

Contextformer 官方论文与仓库使用：

- NDVI prediction NetCDF；
- `s2_dlmask < 1`；
- SCL 类别属于 `{1,2,4,5,6,7}`；
- `landcover < 41`；
- target NDVI 最小值大于 0；
- 未来至少 10 个有效观测；
- 完整序列与预测序列之间至少 3 个额外历史/缺失观测条件；
- target NDVI 标准差大于 0.1；
- 先按 minicube×land-cover 聚合，再平衡不同 land-cover。

仓库中的 `eval/greenearthnet_protocol.py` 已按官方 evaluator commit：

```text
a0329636631371a4aaa9a95c75ed0a37d27b8c4f
```

实现了：

- 时间戳检查；
- official clear mask；
- NDVI；
- R²/RMSE/NSE/bias；
- RMSE25；
- land-cover-balanced aggregation；
- Outperformance 的阈值规则；
- streaming Parquet 汇总。

官方来源：

- 论文：<https://openaccess.thecvf.com/content/CVPR2024/html/Benson_Multi-modal_Learning_for_Geospatial_Vegetation_Forecasting_CVPR_2024_paper.html>
- 代码：<https://github.com/vitusbenson/greenearthnet>

### 5.3 当前同协议链还没有正式闭环

`eval/export_greenearthnet_predictions.py` 当前至少存在以下缺口：

1. **没有调用** `prepare_stage2_batch_for_model`，physical4 正式配置的 deferred resize 仍会不一致；
2. CLI 只有 `--dgh-stats-path`，没有当前 physical4 需要的 `--conditioning-stats-path`；
3. 没有复用 `verify_checkpoint_contract`；
4. 没有 prediction manifest、checkpoint/config/manifest hash；
5. 允许已有输出被跳过，但没有证明旧文件与当前 checkpoint 相同；
6. 默认 `split=ood-t`，而本文冻结 manifest role 是 `ood`，映射需要显式固定；
7. 只通过合成测试验证公式，尚未在真实 cube 上与官方 `eval.py` 做数值 parity；
8. 当前仓库没有生成 Persistence/Climatology NetCDF 和 per-pixel score 的正式入口；
9. 没有 5 日粒度 R²/RMSE 曲线和 tile-cluster bootstrap。

因此当前正确结论是：

> **同协议指标核心已实现，但 physical4 端到端导出、baseline 和真实 parity 尚未完成。**

在上述问题解决前，不能把新算出的 R²/RMSE 与 Contextformer 公开值标成严格公平比较。

## 6. 如何最省时间地使用别人论文结果

### 6.1 推荐主表结构

#### Block A：本文冻结协议、由我们实际计算

| 方法 | 来源 | 同一 manifest/mask/evaluator | 用途 |
|---|---|---|---|
| Persistence | 我们计算 | 是 | 最低参考 |
| Climatology | 我们计算 | 是 | 季节性参考 |
| Direct-P4 | 我们训练与评估 | 是 | 配对结构对照 |
| Rollout-P4 | 我们训练与评估 | 是 | 主模型 |

#### Block B：公开论文参考值

| 方法 | 来源 | 标记 |
|---|---|---|
| Earthformer | 论文公开值 | † |
| PredRNN | 论文公开值 | † |
| SimVP | 论文公开值 | † |
| Contextformer | 论文公开值 | † |

如果我们最终证明自己的预测是在**完全相同的 GreenEarthNet 官方测试集和 evaluator**上产生的，则 Block B 可以和我们的 GreenEarthNet-protocol 行进行数值比较，不需要重训外部模型。

如果 exact parity 没有证明，则 Block B 只能叫“Published Reference（公开参考）”。

### 6.2 200 epoch 对这里没有决定作用

是否能直接用别人论文数值，取决于测试协议，不取决于我们是不是 200 epoch。

200 epoch 主要用于：

- 保证 Direct/Rollout 训练预算充足；
- 保证内部结构对照匹配；
- 报告本文自己的训练成本。

### 6.3 当前最现实的省时方案

## 8. U 在科学上究竟要证明什么

### 8.1 no-U 的主线

```text
过去 50 天真实卫星观测
  → I：初始化内部状态
  → F：在未来 physical4 驱动下每 5 天推进
  → O：解码出 day5...day100 预测
```

这条线证明：

> 模型能否维护一个受外部驱动、可递推、可由未来卫星图像验证的预测状态。

### 8.2 U 的升级

```text
预测到 day25
  → 先输出并评分 day25 prior prediction
  → day25 的部分真实观测到达
  → U 根据“观测－预测”残差修正内部状态
  → 从修正后的 posterior 预测 day30...day100
```

U 要证明的不是“看了答案以后当前帧更准”，而是：

> 在不修改已经完成的当前预测前提下，新观测能否修正模型对世界状态的认识，并改善后续未来。

### 8.3 U 不是不确定性

| 概念 | 问题 |
|---|---|
| F（预测） | 未来会怎样？ |
| uncertainty（不确定性） | 我有多大把握？ |
| U（更新） | 重新看到现实后，如何改正内部状态？ |

不确定性未来可以用于决定是否信任观测或何时更新，但当前 U 的定义是 state update（状态更新），不是 uncertainty estimation（不确定性估计）。

## 10. U 正式进入论文前必须补齐的代码要求

### U-P0：从冻结 Rollout-P4 初始化

必须新增一种与普通 `resume_from` 不同的语义，例如：

```text
initialize_correction_from_stage2 = <selected Rollout-P4 checkpoint>
```

它应当：

1. 加载 Rollout 的 `core.*` 和 `transition.*`；
2. 允许 U 新参数在源 checkpoint 中不存在；
3. 严格拒绝其他非 U 参数的 missing/unexpected keys；
4. 记录 parent checkpoint 路径、SHA256 和 selected checkpoint JSON；
5. 不加载 Rollout 的旧 optimizer/scheduler 作为 U optimizer；
6. 不把 forecast mode 不同误判为普通 exact resume。

当前 `resume_from` 会 strict 加载整个模型、optimizer、scheduler，并检查 curriculum/forecast mode；它不适合把普通 Rollout checkpoint 直接变成带新增 U 参数的模型。

### U-P0：冻结主干，隔离 U 的贡献

第一版正式 U 实验建议：

```text
冻结 I：encoder / phi_encoder / state_projector / context aggregator
冻结 F：driver encoder / horizon encoder / state dynamics
冻结 O：observation decoder
只训练 U：correction cell
```

否则当前 optimizer 会把所有 `requires_grad=true` 的 transition、decoder 等一起训练。最终即使性能提升，也无法证明是 U 的贡献。

如果以后要联合微调，必须作为独立消融：

```text
U-only（主结论）
U + trunk fine-tune（扩展）
```

不能把二者混成一个结果。

### U-P0：固定 reveal 协议

训练可以保留随机 reveal，正式评估必须至少支持：

```text
no reveal
reveal exactly at day25（index 4）
reveal exactly at day50（index 9）
```

并保证所有策略对每个样本使用完全相同的：

- reveal day；
- observation；
- observation mask；
- 有效像元；
- checkpoint base；
- forecast target。

当前 evaluator 使用随机 2–15 step reveal，不能直接填 day25/day50 的正式表。

### U-P0：每样本配对指标

必须保存每个 sample_id 的：

```text
MAE/RMSE at every horizon
No-update error at every horizon
U error at every horizon
paired gain = error_no_update - error_u
visible fraction q
reveal day
tile / land-cover / quality stratum
```

然后计算：

- day25/day50 后绝对 RMSE；
- 每个 horizon 的 paired gain；
- Gain-AUC；
- gain 持续到多少天；
- tile-cluster bootstrap 95% CI；
- no-reveal 最大绝对差。

当前 evaluator 是 batch aggregate，不保存 per-sample paired record，因此不够。

### U-P0：no-reveal 严格一致性

必须显式验证：

```text
同一 selected Rollout checkpoint
+ reveal_mask 全零
→ U wrapper 输出与原 Rollout 输出在容差内相同
```

推荐同时报告：

```text
max_abs_prediction_difference
mean_abs_prediction_difference
metric_difference
```

如果无观测时 U 自己改变预测，就不能声称它是“只在新观测到达时校正”。

### U-P1：observation mask 协议

当前 correction schedule 直接使用训练 `target_mask` 作为 `observation_mask`。论文正式协议要明确：

- 是只用 `s2_mask`；
- 还是 `s2_dlmask + SCL`；
- 是否只在植被 eligible pixels 上更新；
- 训练 mask 与正式评价 mask 是否必须分离；
- 云遮挡比例 q 如何计算。

建议原则：

> observation mask 表示“模型此刻真正看到了哪里”；evaluation mask 表示“哪些位置用于最终评分”。两者语义不同，不能偷用评估 eligibility 帮助模型更新。

### U-P1：VanillaFilter 公平性

VanillaFilter 必须：

- 从同一个 frozen Rollout 初始化；
- 使用同一 reveal 训练数据；
- 主干冻结规则相同；
- 参数量/FLOPs 与 U 相近，建议误差在 ±5%；
- 独立选择 checkpoint；
- 不能拿随机初始化、未训练 filter 当基线。

### U-P1：provenance

U 正式结果文件至少记录：

- U checkpoint SHA256；
- parent Rollout checkpoint SHA256；
- U/VanillaFilter config SHA256；
- train/val/test manifest SHA256；
- conditioning stats SHA256；
- reveal protocol；
- observation mask protocol；
- strategy；
- seed；
- code commit；
- per-sample record hash。

当前 U evaluator 只记录路径和 seed，没有复用完整 Stage2 contract verification/provenance。

## 16. AAAI 写作与审稿风险检查

### 主线

```text
任务：长期地球可观测表面预测
→ 缺口：固定多时距预测不一定维护可递推状态
→ 挑战：长期误差累积、外生驱动和部分可观测
```
