## 0. 一页结论

### 0.1 文章类型

本文定位为**单模型、方法型 world-model 论文**，不是纯分析论文，也不是新 benchmark 论文：

1. 先提出一个观点：天气驱动的 EO 世界模型不应只输出一个固定时域未来，还应形成一个可被推进和复用的预测状态。
2. 针对该观点提出一个模型：**ObsWorld**。
3. 用三类互补证据评价它：公共预测精度、外部天气响应、内部状态契约。
4. 用 matched-backbone 消融证明能力来自 ObsWorld 的状态—转移设计，而不是只来自更大骨干或换评测口径。

最终论文只出现**一个主方法 ObsWorld**。方案 A / B 是研发阶段的两种候选实现，全文截止前二选一；落选者只作为 matched baseline、消融或附录工程探索，不作为第二个并列方法。

### 0.2 当前核心主张

> **我们提出 ObsWorld：一个由天气驱动、具有共享转移的预测状态世界模型；它在公共预测基准上保持充分竞争力，同时支持固定时域预测器无法保证的跨时域推进、天气响应和状态复用。**

注意：这是当前的**目标性主张**。其中“保持充分竞争力”“支持……”都必须由最终结果兑现后，才能在摘要中改成事实语气。

### 0.3 最精确的科学问题

> **一个天气条件化的高分辨率 EO 预测器，能否学习出这样的空间预测表示：它真实承载未来预测，能够在共享的天气驱动转移下跨时距推进，并在不同推进路径下保持非平凡的一致性？**

这就是本文所研究的“什么样的表示和动力学”：

- 表示不是任意 backbone feature，而是**承载预测的空间 predictive state**；
- 动力学不是每个 horizon 各用一个黑盒头，而是**共享、天气条件化、时距条件化的状态转移**；
- “可复用”不是靠命名，而由 load-bearing、driver sensitivity、non-collapse、direct/composed consistency 和 endpoint accuracy 联合检验。

## 1. 世界模型定位：不否定 EO-WM，也不做真假世界模型裁判

### 1.1 正确表述

EO-WM 明确把自身表述为一个世界模型，我们对此没有异议。它将 EO forecasting 视为**部分可观测、天气驱动的 world-modeling problem**，以视频扩散模型和物理化天气条件建模未来多光谱观测。本文不应写“EO-WM 或其他 EO 模型不是世界模型”。

更稳妥的说法是：

> 现有 EO forecasters 和 EO world models 已经能够在稀疏观测与天气 forcing 下预测未来；本文进一步研究，固定时域输出精度之外，模型是否形成了可被推进、可在改变驱动后检验响应并可复用的预测状态。

因此，“world model”不是本文用来排除别人的二元标签，而是一组可以逐项取证的功能主张。

### 1.2 ObsWorld 的最小模型合同

令历史 EO 观测为 $o_{\le t}$，未来天气为 $u$，静态地理背景为 $g$，预测跨度为 $h$：

$$
z_t=q(o_{\le t}),\qquad
z_{t+h}=T(z_t,u_{t:t+h},g,h),\qquad
\hat{o}_{t+h}=O(z_{t+h}).
$$

其中：

- $q$：从受云遮挡的历史观测推断空间预测状态；
- $T$：在未来天气、地理背景和经过时间条件下推进状态；
- $O$：从推进后的状态产生未来观测或 NDVI；
- 同一个 $T$ 同时用于 direct 与 composed 路径。

EO 中没有 agent action 不妨碍 world model 定位；天气在这里承担**外生 forcing**。但必须诚实写成 weather-conditioned / weather-driven，不写 causal intervention 或自主物理模拟。

### 1.3 当前不作为核心的“观测感知”轴

L1C/L2A 因子化是有价值的附加能力，但只有在 product-aware renderer 与最终预测状态真正合一后，才能进入标题和核心贡献。当前主线不依赖它，避免出现：

- 方案 A 的最终预测状态尚未合一，却在摘要中声称 observation-aware；
- 方案 B 的独立 factorizer 很强，却与 forecasting state 是两套状态；
- 为保旧摘要而让一个未闭环的次要实验绑架整篇论文。

因此，当前核心是 **predictive state + shared weather-driven transition**；observation formation 是通过整合门后才升级的第二轴。

## 2. 与现有数据严格对齐的故事：我们到底帮助什么

### 2.1 不能讲的故事

本文数据是天气条件化的植被/地表观测预测，不能因为“世界模型”三个字就扩张成：

- 洪水预测；
- 通用灾害决策；
- 作物产量预测；
- 因果干旱归因；
- 任意气候情景下的数字孪生。

这些任务没有被当前标签、协议和实验直接覆盖。

### 2.2 可以扎实讲的故事

> 在云遮挡、观测稀疏的条件下，根据历史卫星影像和未来天气 forcing，预测未来植被状态。常规固定时域指标衡量“这一次实现的未来像不像”，但真实的植被监测还关心：模型能否在不同 lead time 上重复使用同一状态，能否对热/旱天气压力产生方向和幅度合理的响应，以及分段推进时是否得到相容的未来。

这个故事与数据直接对齐：

1. **GreenEarthNet OOD-t**：检验时间分布偏移下的普通植被预测能力，防止模型“有机制、没精度”。
2. **EarthNet2021 Extreme Summer**：检验热/旱压力下植被下降的谷值误差和下降幅度误差；我们只能称“极端夏季植被压力响应”，不扩张成通用干旱预测系统。
3. **EarthNet2021 Seasonal Matched-Pair**：检验同地点、相近季节初始条件、不同已实现天气 forcing 下，预测差异的方向、幅度和排序是否正确。
4. **ObsWorld 状态检验**：解释为什么模型能或不能完成上述行为，并检验同一状态能否跨时距推进。

### 2.3 预测状态真正服务的能力

如果证据通过，同一个状态—转移接口可以服务：

- **可变时域预测**：同一状态推进到多个 lead time，而非每个 horizon 独立建头；
- **天气条件化情景响应**：在相同初始状态下输入不同天气序列，比较预测响应；
- **稳定滚动**：直接走 $h$ 与分段走 $h_1+h_2$ 不产生互相矛盾的未来；
- **状态复用**：在确有 frozen-readout 实验时，用简单头读取未来植被衰退等事件。

证据边界：当前 observational benchmark 最多支持 **response fidelity / scenario-conditioned simulation**。在没有干预数据和识别假设时，不写“因果反事实正确”。

### 2.4 从动机到结论的完整因果链

```text
稀疏、受云遮挡的 EO 观测 + 已知未来天气
                    ↓
固定时域像素精度无法判断模型是否依赖天气、是否可跨时距复用
                    ↓
需要一个承载预测的空间状态 + 共享天气驱动转移
                    ↓
提出单一模型 ObsWorld（q → T → O）
                    ↓
公共预测精度 + 外部天气响应 + 内部状态契约
                    ↓
回答：准确 EO forecasting 何时真正支持可复用 world modeling
```

## 3. 方案 A / B 是否都符合当前叙事

### 3.1 总结判断

**同一个科学问题和摘要包络可以覆盖 A/B；但当前两个实现对强主张的完成度不同。**

| 项目 | 方案 A：ViT-S direct-state 路线 | 方案 B：PVT/Contextformer B4 路线 |
|---|---|---|
| 历史观测编码 | Stage1.5 ViT-S | PVT/Contextformer |
| 公共预测潜力 | 当前仍需提升 | 当前 matched backbone 更强 |
| 显式状态与转移 | `z0 → T(D,G,h) → zh → O` | 投影状态 `s` + `T(s, weather)` |
| 转移后状态是否进入最终预测 | **当前设计是** | **当前 B4 代码不是：forecast 保持 B0 路径，T/O 是 auxiliary heads** |
| direct/composed 端点预测 | 架构上天然可做 | 当前需增加从转移状态到预测的读出/残差路径 |
| 当前叙事角色 | 结构闭环更干净、精度风险较高 | 精度底座更强、load-bearing 闭环尚需补齐 |

### 3.2 为什么不能简单说“无论 A/B 当前形态都完全一样”

方案 B 当前 `models/plan_b_b4.py` 明确保持 B0 的 forecast path 不变，新增的 projector、transition 和 renderer 以 auxiliary loss/capability head 形式存在。这有利于恢复强基线，但也意味着：

- 关掉 $T$ 不会在推理时改变 B0 预测；
- $T(s)$ 当前不能自然产生 direct/composed 的 endpoint forecast；
- 因而不能直接声称“同一推进状态承载最终预测”。

这不代表 B 不能成为最终 ObsWorld，而是要求最终选择 B 前完成以下至少一种闭环：

1. 用 $T(s)$ 解码未来预测，并让该路径承担主要或可测的预测增量；或
2. 将 state-decoded prediction 作为 zero-initialized residual 接回 B0 输出，既保持初始可恢复性，又让训练后状态转移真实影响结果；或
3. 给出严格干预证据，证明所定义的同一个 state 是 forecast-bearing，且 direct/composed 都从它产生有效 endpoint。

### 3.3 A/B 最终二选一规则

现在不选。最终候选必须同时过：

- **P0 协议门**：相同数据、预处理、预测长度、evaluator；
- **P1 精度充分性门**：公共 OOD-t 结果不崩盘，并报告逐 minicube 配对差异；
- **P2 load-bearing 门**：预测必须真实依赖所声称的状态和转移，不得旁路；
- **P3 driver 门**：matched weather 优于 shuffled/null weather，且不是只在 latent 中发生无输出影响的变化；
- **P4 composition 门**：direct/composed 两路各自 endpoint 合格，再谈一致性；
- **P5 anti-collapse 门**：状态变化量、方差/有效秩、shuffle 下限排除恒等或坍塌；
- **P6 matched-backbone 门**：ObsWorld 相对自身相同预算底座不出现不可接受退化。方案 B 现有用户硬约束是 B4 必须优于 B0。

通过后，胜出的候选统一命名为 **ObsWorld**；骨干只写在实现细节中。

## 4. 我们研究的“表示”到底是什么

### 4.1 一句话定义

> **ObsWorld 研究的是一种 forecast-bearing spatial predictive state：它由受云遮挡的历史观测推断，包含完成未来地表预测所需的信息，并由同一个天气条件化转移跨时距推进。**

“包含所需信息”当前只表示经验上的 future-predictive；除非完成 screening-off 检验，不写 sufficient statistic。

### 4.2 六个可证伪属性

| 属性 | 问题 | 最小证据 |
|---|---|---|
| Forecast-bearing | 状态是否真实承载预测？ | 切断/打乱状态或转移会显著损害同一路径预测；无旁路 |
| Future-predictive | 状态是否包含未来相关信息？ | transitioned state 对真实 future latent / endpoint 有预测力 |
| Driver-sensitive | 状态是否响应天气？ | matched vs shuffled/null weather，状态与输出均有差异 |
| Non-degenerate | 好看的一致性是否来自恒等映射？ | state delta、维度方差、有效秩、跨样本归一化 |
| Path-consistent | 同一时间间隔不同分解是否相容？ | direct vs 3+7、4+6、5+5 等多 partition |
| Reusable | 同一状态是否支持多查询？ | 多 horizon；可选 frozen event readout；外部天气响应协议 |

### 4.3 不要把“表示研究”误写成表示学习大而全

本文不比较所有 EO foundation model 表示，也不声称恢复真实完整物理状态。核心是：

> **在同一个预测任务中，怎样的显式状态—转移设计，能让强预测器同时暴露出可检验、可推进的预测表示？**

这是一个方法问题，而不是 feature visualization 论文。

## 5. 三层实验尺子与数据准备

### 5.1 三层证据，不是换掉不利尺子

| 层次 | 协议 | 证明什么 | 不能证明什么 |
|---|---|---|---|
| Table 1 | GreenEarthNet OOD-t | 标准预测能力、时间 OOD 充分性 | 状态可组合、天气响应正确 |
| Table 2（可选但推荐） | EO-WM 在原始 EarthNet2021 上定义的 Extreme / Seasonal | 极端夏季植被下降与差异天气下的输出响应 | 内部状态真的承载预测；因果反事实 |
| Table/Figure 3 | ObsWorld matched ablations + state diagnostics | load-bearing、non-collapse、driver sensitivity、composition | 社区通用 benchmark 排名 |

正确措辞是“增加正交证据”或“采用独立定义的外部行为协议”，不是“离开 Contextformer 的尺子去有利战场”。GreenEarthNet 仍是必须通过的公共精度门。

### 5.2 EO-WM 的 Extreme / Seasonal 是否要额外数据

结论分两层：

1. **不需要另找一个新的干旱或灾害数据集，也不需要人工制作干旱标签。** EO-WM 的 CSV 已给出样本窗口/配对及元数据，TN-MAE、DAE、DRR、DHR、PDC 从原始卫星与天气数据派生。
2. **若要与 EO-WM 论文数字直接同表，需要额外准备它指定的原始 EarthNet2021 测试文件。** 官方仓库要求下载 `extreme` 和 `seasonal` split，路径为 `extreme_test_split/.../*.npz` 与 `seasonal_test_split/.../*.npz`。已发布元数据包含 1,440 个 Extreme Summer 窗口和 422 个 Seasonal Matched-Pair（844 个推理窗口）。

因此：

- 如果服务器已经有原始 EarthNet2021 NPZ 的 extreme/seasonal，则不需再下载；
- 如果只有 GreenEarthNet / `earthnet2021x` 的 NetCDF，则还缺 exact benchmark 所需原始测试文件；
- 只需先下载测试 split 做 zero-shot 适配评估，不必立即增加新训练数据；
- 但训练数据版本、天气变量、归一化与 10→20 窗口适配必须如实披露并核验。

EO-WM 官方仓库明确说明：发布物只包含 benchmark metadata 和评测示例，模型输入代码是 Earthformer-specific，但指标逻辑可以适配任何能够生成对齐 EarthNet2021 target-window prediction 的模型。

EO-WM 截至本文件日期仍是预印本。因此它可以作为**独立、同期、已公开协议**，但不能写成已被顶会验收的社区标准或先例。

### 5.3 能不能只用自己的 GreenEarthNet 数据完成类似实验

可以做“类似目的”的内部诊断，但证据等级不同：

- 在 GreenEarthNet 中筛 extreme cases / matched weather pairs：可以支持内部分析；
- 不能把它与 EO-WM Table 2 数字直接比较；
- 如果样本筛选、阈值和指标均由我们看过结果后决定，会有 cherry-picking 风险；
- 这会更接近“我们自己造了一把尺子”，反而增加 benchmark 质疑。

因此首选顺序是：

1. 原样采用 EO-WM 已发布 CSV + 原始 EarthNet2021 NPZ；
2. 只写模型 adapter，不改样本定义与指标；
3. GreenEarthNet 自定义诊断只作补充/附录，并预先冻结规则。

### 5.4 为什么这一外部评测与本文叙事合理

- 研究对象仍是植被/地表预测，不跨到洪水等别的领域；
- Extreme Summer 检验热/旱压力下的植被下降，直接对应天气驱动状态；
- Seasonal Matched-Pair 检验不同已实现天气条件下预测差异的方向与幅度；
- 两者是输出层行为证据，ObsWorld 状态实验是内部机制证据，互不替代；
- 这是外部 concurrent work 定义的协议，降低“为自己定制指标”的嫌疑。

### 5.5 重要语言边界

- 写 `extreme-summer vegetation response`，不直接写 `drought forecasting`；
- 写 `response under differing realized weather forcing`，不写 `causal counterfactual`；
- DHR/PDC 是输出行为指标，不等于 latent composition；
- GreenEarthNet 与原始 EarthNet2021 的数值不混表；
- 本地 GreenEarthNet val NDVI-MAE 不能与 EarthNet2021 extreme N-MAE 直接比较。

## 6. 为什么本文不会演化成 benchmark 论文

增加尺子不等于变成 benchmark。论文是否是 benchmark paper，取决于贡献中心，而不是表格数量。

### 6.1 方法论文的四条硬约束

1. 标题和摘要先写 **ObsWorld 方法**，不写“a new benchmark for …”。
2. 不发布新数据集、不建立 leaderboard、不把样本构造作为第一贡献。
3. EO-WM 的 Extreme/Seasonal 按外部协议原样使用，称为 evaluation protocol / diagnostic benchmark。
4. 我们自己的状态指标用于检验模型合同，并通过 matched ablation 解释方法，不包装成社区通用综合排行榜。

### 6.2 贡献顺序必须保持

1. **观点/形式化**：fixed-horizon accuracy 不足以保证 reusable predictive state；
2. **方法**：一个 forecast-bearing spatial state + shared weather-driven transition 的 ObsWorld；
3. **实证**：在公共精度、外部天气响应和内部状态合同上验证；
4. 可复现评测 adapter 只是工程产物，不单独升格为主要贡献。

### 6.3 需要警惕的滑坡信号

若出现以下情况，说明论文正在滑向纯 benchmark，应立即纠偏：

- 主要篇幅都在解释指标和样本筛选，模型只有一小段；
- 没有 matched backbone 或方法消融；
- ObsWorld 的状态转移不参与预测，只剩“给别人模型打分”；
- 标题开始使用 `benchmarking` / `evaluation suite`；
- 结论只能说“现有模型在某指标失败”，不能说 ObsWorld 为什么更好。

## 7. 标题、TL;DR 与摘要（中英文）

### 7.1 推荐标题

**English**

> **ObsWorld: A Testable Predictive-State World Model for Weather-Driven Land-Surface Forecasting**

**中文**

> **ObsWorld：面向天气驱动地表预测的可检验预测状态世界模型**

为什么用这一版：

- `ObsWorld` 明确是一个方法；
- `predictive-state world model` 指出方法身份；
- `testable` 表示状态主张接受证伪，而不是预先宣称已经 composable；
- `weather-driven land-surface forecasting` 与现有数据完全对齐；
- 不绑定 ViT-S / PVT，最终选 A 或 B 都不改变研究问题。

若最终 composition 多分段、anti-collapse、endpoint guard 全部很强，可在全文阶段考虑更强标题：

> *ObsWorld: Learning Composable Predictive States for Weather-Driven Land-Surface Forecasting*

在结果未定前不推荐该强标题。

### 7.2 TL;DR

**English**

> ObsWorld learns a forecast-bearing spatial state advanced by a shared weather-conditioned transition, and tests whether accurate land-surface forecasting also supports reusable, driver-sensitive, and path-consistent world modeling.

**中文**

> ObsWorld 学习一个由共享天气条件化转移推进、真实承载预测的空间状态，并检验准确地表预测是否同时支持可复用、受驱动且路径一致的世界建模。

### 7.3 中文摘要（主线安全版）

> 天气条件化的地球观测预测根据稀疏、受云遮挡的历史影像与气象驱动预测未来地表观测。然而，模型通常仅按固定时域像素精度选择，这不能说明其内部表示是否可作为可复用的世界状态：模型可能忽略天气、退化为接近持续性预测，或在直接与分段跨越同一时间区间时到达不相容的状态。我们提出 **ObsWorld**，一个面向高分辨率地表预测的预测状态世界模型。ObsWorld 从历史观测推断空间状态，并通过由未来天气、地理背景和经过时间共同条件化的共享转移推进它；最终预测必须由该状态承载，同一状态与转移同时支持直接和组合预测路径。我们从三个互补维度评价 ObsWorld：GreenEarthNet 上的时间分布外预测、不同天气条件下的响应保真度，以及针对非退化性、天气敏感性、状态承载性和路径一致性的受控诊断。预测精度与状态证据分开报告，并通过同骨干、同预算的消融排除单纯架构增益。**【最终结果句】** 本研究由此检验：一个准确的 EO 预测器何时也能提供用于可变时域、天气条件化模拟的可复用动力学状态。

### 7.4 English abstract (registration-safe)

> Weather-conditioned Earth-observation forecasting predicts future land-surface observations from sparse, cloud-obscured image histories and meteorological drivers. Yet models are usually selected by fixed-horizon pixel accuracy, which does not establish whether their internal representations can serve as reusable world states: a model may ignore weather, remain close to persistence, or reach incompatible states when the same interval is traversed directly and compositionally. We introduce **ObsWorld**, a predictive-state world model for high-resolution land-surface forecasting. ObsWorld infers a spatial state from observation histories and advances it with a shared transition conditioned on future weather, geographic context, and elapsed time. The state is required to be forecast-bearing, while the same state and transition support direct and composed prediction paths. We evaluate ObsWorld along three complementary axes: temporal out-of-distribution forecasting on GreenEarthNet, response fidelity under differing weather conditions, and controlled diagnostics of non-degeneracy, weather sensitivity, load-bearing use, and path consistency. Forecast accuracy is reported separately from state evidence and guarded by matched-backbone, matched-budget ablations. **[FINAL RESULT SENTENCE]** Our study therefore tests when an accurate EO forecaster also provides a reusable dynamical state for variable-horizon, weather-conditioned simulation.

## 8. 论文贡献的最终写法

### 8.1 三条贡献

1. **问题与形式化**：指出固定时域预测精度不能识别 forecaster 是否形成可复用世界状态，并将天气驱动 EO world modeling 形式化为 forecast-bearing predictive state 与共享转移的联合学习问题。
2. **方法**：提出 ObsWorld，通过显式空间状态、未来天气/地理/时距条件化的共享转移，以及 direct/composed 共用接口，把预测与状态动力学置于同一模型中。
3. **实证**：在公共 OOD 预测、独立天气响应协议（若完成）和带 anti-collapse 护栏的状态检验上，用 matched-backbone 消融验证精度、驱动响应和状态复用之间的关系。

### 8.2 不属于本文贡献的内容

- 不声称首次提出 EO world model；
- 不否定 EO-WM、VegSim 等模型的 world-model 身份；
- 不提出通用 EO benchmark；
- 不声称 causal、digital twin 或完整物理状态；
- 不声称洪水/产量等未评测任务的直接贡献；
- L1C/L2A 未合一前，不把 observation-aware 放入标题/首要贡献；
- 未做 screening-off 前，不写 sufficient statistic。

## 9. 实验设计最小闭环

### 9.1 Table 1：公共预测充分性

至少报告：

- GreenEarthNet 官方 $R^2$、RMSE、NSE、absolute bias、Outperformance-vs-Climatology、RMSE25；
- matched backbone；
- Contextformer / PredRNN / SimVP / Earthformer 等公开参照，清楚区分 published 与 locally reproduced；
- 逐 minicube `Win / Tie / Loss` 与 cluster bootstrap CI；
- 若推广 GreenEarthNet 有效差异阈值，明确写成“our paired generalization”，不冒充官方 aggregate equivalence。

GreenEarthNet 的原始逐 minicube Outperformance 规则是：四项指标中至少三项达到有效改善；RMSE / absolute bias 的差异阈值为 0.01，NSE / $R^2$ 的差异阈值为 0.05。我们若将同一规则推广为 `W/T/L vs Contextformer`，必须在表注中说明这是**对官方配对规则的自定义推广**，不能据此宣称全局均值统计等价。

### 9.2 Table 2：外部天气响应（可选但推荐）

- Extreme Summer：ENS、P-MAE、N-MAE、TN-MAE、DAE，含 severity bins；
- Seasonal Matched-Pair：DRR、DHR、PDC；
- 必须使用发布 CSV、原始 EarthNet2021 split 与冻结 metric code；
- 不把 GreenEarthNet val 数字与该表混合；
- 不把输出层 DHR/PDC 解释成 latent composition。

### 9.3 Table/Figure 3：方法机制证据

- matched backbone vs ObsWorld；
- state/T intervention；
- matched vs shuffled/null weather；
- direct vs 3+7、4+6、5+5 等 composed paths；
- 两路 endpoint forecast accuracy；
- state delta / state std、有效秩、跨样本归一化；
- 可选 frozen event readout，但没完成就不把 downstream reuse 写成已证实结果。

### 9.4 最关键的结果组合

理想结果不是每一列都第一，而是：

1. Table 1 证明 ObsWorld 没有为机制牺牲到不可用；
2. Table 2 证明它对天气变化和极端夏季植被下降更忠实；
3. Table/Figure 3 证明这种行为来自承载预测、受天气驱动且可组合推进的状态；
4. matched ablation 证明不是“换骨干/多数据所以更好”。

## 10. 审稿质疑与有力应答

### 10.1 “内部状态这些性质没有意义”

弱回答：世界模型本来就应该这样。  
强回答：

> 固定时域精度相近的模型可能在替换天气、改变 lead time 或递归推进时表现完全不同。load-bearing、driver sensitivity 和 composition diagnostics 不是审美要求，而是在标准误差看不见时，识别这些部署相关失败模式的机制检验。我们进一步用外部天气响应协议和 endpoint guard 证明指标与可观察行为相连。

### 10.2 “你只是精度不行所以换尺子”

应答必须基于实验设计：

- GreenEarthNet 主表不删除、不隐藏；
- 外部协议来自 concurrent EO-WM，不是看过我们结果后自造；
- 状态指标预先冻结并带 anti-collapse；
- 所有能力与 matched strong backbone 同预算比较；
- 若精度崩盘，论文不靠机制指标宣布成功。

### 10.3 “这不就是一个 benchmark paper？”

> 不是。本文的主要产物是一个带显式预测状态和共享天气转移的模型。外部协议仅用于验证模型声称的天气响应，内部 diagnostics 用于方法消融；我们不提出新数据集、排行榜或以 benchmark construction 为贡献中心。

### 10.4 “为什么叫 world model？”

> 因为模型不是只拟合固定输出，而是显式推断环境预测状态，以外生天气 forcing 推进该状态，并从推进状态产生未来观测；此外，这三个接口分别接受 load-bearing、driver 和 composition 检验。我们不以名称替代证据，也不声称恢复完整物理世界。

## 11. 本轮九个问题的正式问答记录

### Q1：EO-WM 本来就是世界模型，我们是不是表述错了？

**是表述需要纠正，不是研究方向需要推翻。** EO-WM 明确是世界模型。我们不判断它“算不算”，而是在其基础上追问：除固定时域输出与天气响应外，内部状态是否真实承载预测、可共享推进和组合复用。

### Q2：最终在方案 A/B 中二选一，两者都符合现在叙事吗？

**科学问题、标题和摘要包络都兼容 A/B。** 但强方法主张有共同硬门：最终状态转移必须真实进入预测。A 当前天然满足；B 当前 B4 的 T/O 仍是 auxiliary，若最终选 B，需要先补 load-bearing 路径。全文只保留胜出的一个 ObsWorld。

### Q3：我们的能力到底服务什么样的、与数据一致的故事？

服务于**天气驱动的植被状态预测**：多 lead time 推进、热/旱极端夏季下植被下降响应、不同天气 forcing 下预测差异，以及稳定分段滚动。不能扩张到洪水、通用灾害或因果干旱预测。

### Q4：以方案 A 为例，我们到底研究什么表示？

研究由云遮挡历史推断的 **forecast-bearing spatial predictive state**。它不是普通 ViT feature，而是通过未来预测、天气扰动/置换、状态非坍塌和 direct/composed 检验共同定义的预测表示；相应动力学是共享的 $T(z,D,G,h)$。

### Q5：极端天气响应是否需要额外数据？设计是否合理？

不需要另找新领域或新标签数据集；但 exact EO-WM comparison 需要原始 EarthNet2021 `extreme`/`seasonal` NPZ 测试 split。如果当前只有 GreenEarthNet NetCDF，就需要额外下载这些测试文件。设计合理，因为任务仍是植被预测，且协议独立定义；本文件已完整记录，采用与否后续再决定。

### Q6：增加/更换尺子会不会把论文变成 benchmark？

不会，只要 GreenEarthNet 不被替换、ObsWorld 方法和 matched ablation 始终居中、外部协议只作验证，并且不提出新数据集/leaderboard。危险的不是多一张表，而是模型不再承重、全文只剩给别人打分。

### Q7：标题和摘要是否有中文版，并落实到文件？

已在 §7 给出中英文标题、TL;DR、主线安全版摘要、可选外部协议句和结果句模板。

### Q8：是否保留“由天气驱动、共享转移……”这句话？

保留，作为全项目的一句话北极星，见 §0.2。最终只有在公共精度、load-bearing、driver 和 composition 证据通过后，才能从目标性主张改为论文结果事实。

### Q9：是否已经把整体主线、约束、易错点整理成专门文件？

即本文件。后续叙事讨论优先回到本文件增补；实验执行细节仍分别保留在方案 A 的 74 号与方案 B 的执行日志中，避免主线文档被命令和临时结果淹没。

## 12. 冻结项、可变项与停止条件

### 12.1 冻结项

- 单模型 ObsWorld；
- 天气驱动 EO / land-surface forecasting；
- forecast-bearing predictive state；
- shared weather/time-conditioned transition；
- 公共精度与状态证据分开报告；
- matched backbone + anti-collapse + endpoint guard；
- 不把他人排除出 world-model 类别；
- 不变成新 benchmark 论文。

### 12.2 可变项

- 最终方案 A 或 B；
- ViT-S / PVT 等骨干；
- physical4 / full weather features；
- transition 的具体参数化与 loss 权重；
- 是否纳入 L1C/L2A observation formation；
- 是否采用 EO-WM external protocols；
- 最终结果句强度。

### 12.3 不能靠摘要措辞掩盖的停止条件

- 公共精度明显崩盘；
- 声称的状态/转移被预测路径绕过；
- direct/composed 一致性来自近恒等或坍塌；
- weather shuffle 对输出无影响；
- 只在自定义有利指标上获胜；
- A/B 都不能同时满足预测充分性与 world-model 合同。

出现这些情况应继续改模型或降低投稿主张，而不是用“world model”包装。
