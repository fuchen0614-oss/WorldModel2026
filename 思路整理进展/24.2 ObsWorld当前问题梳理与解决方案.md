## 1. 技术逻辑问题详解与解决方案

### 问题1：字段归类混乱（P0，最关键）

#### 1.1 问题描述

**原始问题**："NDVI属于状态变量、sun属于成像条件，不应笼统归入外生驱动D"

**问题本质**：你的合作者陈志盛指出，当前22/23文档中字段分类逻辑不一致：
- `ndvi_previous` 被放在 D（外生驱动）中
- `sun_elevation` 同时出现在 D 和 phi 中
- `season` 同时出现在 D 和 phi 中

这会导致：
1. **概念混乱**：到底什么算"驱动"、什么算"状态"、什么算"成像条件"
2. **模型冗余**：同一信息在多个地方重复输入
3. **物理解释性差**：无法清晰说明"模型用了什么条件预测"

#### 1.2 为什么会混乱？

让我先解释每个概念：

| 概念 | 定义 | 例子 | 作用位置 |
|------|------|------|---------|
| **状态变量** | 模型要预测的量，是系统内部状态 | NDVI、土壤湿度、植被覆盖度 | z（潜状态） |
| **成像条件 phi** | 影响"观测外观"但不影响地表本身 | 太阳角、云、传感器、季节（作为拍摄时间） | 编码器/解码器 |
| **外生驱动 D** | 推动状态演化的外部因素 | 降雨、温度（作为生长驱动） | 动力学模块 |

**混乱的根源**：有些量既影响"状态演化"又影响"观测外观"，比如：
- **season**：作为成像条件（不同季节拍摄外观不同）+作为生长驱动（季节推动物候）
- **sun_elevation**：主要是成像条件（影响阴影、辐射计算）
- **ndvi_previous**：这是状态的历史值，不是外生驱动

#### 1.3 正确的分类标准

> [!important] 分类原则
> **按物理过程分，不是按数据来源分。**
> 
> - **phi**：只影响"看起来怎样"，不影响地表物理过程
> - **D**：推动地表状态变化的外部力量
> - **G**：约束状态变化合理性的空间背景
> - **状态反馈**：z_t 本身或从 z_t 解码的量（如 ndvi_previous）

#### 1.4 具体字段重新归类

| 字段 | 错误归类 | 正确归类 | 理由 |
|------|---------|---------|------|
| **ndvi_previous** | ❌ D（外生驱动） | ✅ **状态反馈**（单独类别） | NDVI是状态变量，不是外生驱动。应从 z_t 解码或从 X_t 直接算，作为自回归基线 |
| **sun_elevation** | ❌ D（外生驱动） | ✅ **phi（成像条件）** | 太阳角主要影响成像辐射和阴影，对地表物理过程影响极小 |
| **season** | 🤔 同时在 D 和 phi | ✅ **拆分**：<br>• phi_season（成像时间）<br>• day_of_year（D，物候驱动） | 季节有双重作用，拆成两个字段避免混淆 |
| **temperature** | ✅ D | ✅ D | 正确，推动植被生长、融雪等 |
| **precipitation** | ✅ D | ✅ D | 正确，推动洪水、植被生长 |
| **elevation** | ✅ G | ✅ G | 正确，静态地理背景 |

#### 1.5 解决方案

**立即行动（今天）**：

1. **修改22文档的字段清单**：
   ```
   D 核心字段（纯外生驱动）：
   - precipitation
   - temperature_2m
   - solar_radiation
   - day_of_year（作为物候周期驱动，sin/cos编码）
   
   移除：
   - ❌ ndvi_previous（改为状态反馈）
   - ❌ sun_elevation（移到phi）
   - ❌ season（拆分）
   ```

2. **新增"状态反馈"类别**：
   ```python
   # 在动力学模块输入时
   state_feedback = {
       'ndvi_current': compute_ndvi(z_t),  # 从当前状态解码
       'lulc_current': argmax(z_t_lulc_logits)  # 当前土地覆盖
   }
   # 作为辅助输入，与 D/G/h 并列
   z_pred = Dynamics(z_t, D, G, h, state_feedback)
   ```

3. **phi 和 D 的去重规则**：
   - **season 处理**：phi 中保留 `season_index`（0春1夏2秋3冬），D 中用 `day_of_year`（sin/cos连续编码）
   - **sun 处理**：只在 phi 中保留 `sun_elevation`，D 中不要
   - **原则**：如果一个量主要影响成像，归 phi；主要影响物理过程，归 D；两者都有，拆开

### 问题2：DEM边界偏差（P0）

#### 2.1 问题描述

**原始问题**："小patch内直接计算DEM派生的坡度、坡向等水文特征会产生误差，需要缓冲区/更大范围计算"

**问题本质**：

你的数据是 264×264 像素的小 patch（约 2.64km×2.64km），在这么小的范围内计算水文特征（slope、flow_direction、flow_accumulation）会有问题：

```
问题示意：
┌─────────────────┐
│  小patch边界     │  如果真实水流从patch外流入，
│  ┌──────────┐   │  只在patch内计算flow会丢失
│  │ DEM data │   │  上游信息，导致flow_accumulation错误
│  └──────────┘   │
└─────────────────┘
      ↑ 真实上游在patch外
```

**为什么会有误差**？

1. **坡度/坡向**：边界像素的梯度计算缺少邻域信息（需要周围8个像素，边界缺失）
2. **水流方向**：D8算法需要知道每个像素周围8个方向哪个最低，边界处无法准确判断
3. **汇流累积**：需要追溯上游所有贡献，patch边界截断了上游路径

#### 2.2 解决方案

**方案A：缓冲区计算（推荐）**

```python
# 伪代码
def compute_flow_features(patch_center_lat, patch_center_lon, patch_size=264):
    # 1. 下载更大范围的DEM（patch + buffer）
    buffer = 50  # 额外50像素边界
    dem_large = download_dem(
        lat=patch_center_lat, 
        lon=patch_center_lon, 
        size=patch_size + 2*buffer  # 364×364
    )
    
    # 2. 在大范围上计算水文特征
    slope_large = compute_slope(dem_large)
    flow_dir_large = compute_d8(dem_large)
    flow_acc_large = compute_accumulation(flow_dir_large)
    
    # 3. 裁剪回原始patch大小（去掉buffer）
    slope = slope_large[buffer:-buffer, buffer:-buffer]  # 264×264
    flow_acc = flow_acc_large[buffer:-buffer, buffer:-buffer]
    
    return slope, flow_dir, flow_acc
```

**优点**：
- 边界误差显著降低
- 水流上游信息完整

**缺点**：
- 需要额外下载/存储更大范围DEM
- 计算量增加（但可接受，只需算一次）

**方案B：使用全球水文产品（备选）**

直接下载已经算好的全球水文特征产品：
- **HydroSHEDS**：全球 flow direction、flow accumulation（90m分辨率）
- **MERIT Hydro**：全球水文校正DEM + flow网络（90m）

**优点**：
- 专业产品，精度高
- 无边界问题（全球尺度计算）

**缺点**：
- 分辨率粗（90m vs 你的10m）
- 需要重采样对齐

#### 2.3 立即行动

**今天/明天**：
1. 实现方案A的缓冲区计算脚本 `scripts/compute_flow_with_buffer.py`
2. 对比测试：有buffer vs 无buffer，检查边界处 flow_accumulation 的差异
3. 如果差异显著（>20%），全面采用buffer方案
4. 更新22文档 §7.2"从DEM派生"章节，说明buffer策略

### 问题3：ERA5数据泄露风险（P0，实验严谨性）

#### 3.1 问题描述

**原始问题**："若预测时使用T到T+H期的真实数据，会出现'用未来数据训练'的问题"

**问题本质**：

ERA5提供的是**历史气象再分析数据**，包含过去所有时刻的真实天气。在做时序预测时，必须严格区分：

| 阶段 | 时间 | 可用的ERA5数据 | 不可用的ERA5数据 |
|------|------|---------------|----------------|
| **训练时** | t_current | ✅ [0, t_current] 的历史 | ❌ [t_current, t_future] 的未来 |
| **推理时** | t_current | ✅ [0, t_current] 的历史 | ❌ [t_current, t_future] 的未来真值 |
| | | ✅ [t_current, t_future] 的**预报** | （用天气预报，不是真值） |

**错误做法示例**（会泄露）：
```python
# ❌ 错误：训练时用了未来真实天气
t_current = sample['time_t']          # 2020-06-01
t_future = sample['time_t_h']         # 2020-06-08 (7天后)
D = load_era5(time_range=[t_current, t_future])  # ❌ 包含未来7天真实降雨
z_pred = Dynamics(z_t, D, G, h=7)
loss = ||z_pred - z_real||²
# 问题：模型看到了未来真实降雨，这是作弊
```

**正确做法**：
```python
# ✅ 训练时：用未来观测作为监督，但D只用历史或预报
t_current = sample['time_t']
t_future = sample['time_t_h']

# D可以有两种策略：
# 策略A：只用当前时刻的D（弱驱动）
D_current = load_era5(time=t_current)  # 单时刻
z_pred = Dynamics(z_t, D_current, G, h=7)

# 策略B：用历史统计/气候平均（中等驱动）
D_climate_avg = get_climate_average(location, day_of_year=doy_future)
z_pred = Dynamics(z_t, D_climate_avg, G, h=7)

# 策略C：用天气预报（强驱动，需额外数据）
D_forecast = load_weather_forecast(t_current, horizon=7)  # 从t发布的7天预报
z_pred = Dynamics(z_t, D_forecast, G, h=7)

# 监督：用真实未来观测
z_real = Enc(X_future, phi_future)
loss = ||z_pred - z_real||²
```

#### 3.2 为什么会有这个问题？

因为22/23文档中没有明确说明**D的时间来源规则**。目前只说了"ERA5提供D"，但没说：
- 训练时D是哪个时刻的？
- 推理时D从哪来？

#### 3.3 解决方案

**明确D的时间规则**：

> [!important] D的时间对齐规则
> 
> **原则**：D只能包含"在t时刻已知的信息"
> 
> **训练时**：
> - 选项1（弱驱动）：D = ERA5(t)，只用当前时刻
> - 选项2（中等）：D = 气候平均(location, doy_{t+h})
> - 选项3（强驱动，需额外数据）：D = 天气预报(从t发布的h天预报)
> 
> **推理时**：
> - 实际应用场景用选项3（天气预报）
> - 消融实验可对比选项1/2/3的效果

**立即行动**：

1. **修改22文档§7.4 ERA5章节**，新增"时间对齐规则"小节
2. **修改23文档§3.5训练数据流**，明确代码示例中D的时间来源
3. **在实验设计中明确**：
   ```
   主实验：用选项2（气候平均）
   消融实验：对比选项1（当前） vs 选项2（平均） vs 选项3（预报）
   ```

### 问题4：Stage 2训练策略不明确（P1）

#### 4.1 问题描述

**原始问题**："需要区分主监督目标和辅助监督目标，明确权重关系"

**问题本质**：

23文档§3.3给出了混合loss：
```
L_total = α·L_state + β·L_obs + γ·L_proxy + δ·L_kl
```

但没说：
1. 哪个是"主目标"，哪个是"辅助"？
2. 权重怎么定？
3. 训练策略是什么（阶段性调整？）

#### 4.2 解决方案

**明确loss层级**：

| Loss | 角色 | 权重 | 训练阶段 |
|------|------|------|---------|
| **L_state** | 主监督 | α=1.0（基准） | 全程 |
| **L_kl** | 正则 | δ=0.0001-0.001 | 全程 |
| **L_obs** | 辅助监督 | β=0.1-0.5 | Stage 2b后期加入 |
| **L_proxy** | 辅助监督 | γ=0.05-0.1 | Stage 2c精调时加入 |

**阶段性策略**：

```
Stage 2a（跑通架构，5k steps）：
  L_total = L_state + 0.001*L_kl
  目标：验证动力学模块不崩、loss收敛

Stage 2b（加解码器，20k steps）：
  L_total = L_state + 0.3*L_obs + 0.001*L_kl
  目标：端到端优化观测重建

Stage 2c（精调，10k steps）：
  L_total = L_state + 0.3*L_obs + 0.05*L_proxy + 0.001*L_kl
  目标：对齐NDVI等代理任务
```

**立即行动**：补充到23文档§3.3
